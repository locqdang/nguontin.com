# Data Model: Raspberry Pi PostgreSQL Data Platform

**Feature**: `002-raspi-postgres-data`
**Created**: 2026-07-02

## Scope

This feature does not introduce new business entities. It focuses on translating the existing application data schema from SQLite to PostgreSQL for the Raspberry Pi host.

## Core tables

### users
- `id`: serial primary key
- `email`: text, unique, not null
- `password_hash`: text, nullable
- `role`: text, not null, constrained to `journalist`, `expert`, `admin`
- `full_name`: text, nullable
- `auth_preference`: text, not null, default `email_login`
- `email_verified_at`: timestamptz, nullable
- `verification_status`: text, not null, default `not_started`
- `verification_method_label`: text, nullable
- `created_at`: timestamptz, not null, default CURRENT_TIMESTAMP
- `updated_at`: timestamptz, not null, default CURRENT_TIMESTAMP

### email_login_challenges
- `id`: serial primary key
- `email`: text, not null
- `code_hash`: text, not null
- `intended_role`: text, nullable, constrained to `journalist`, `expert`, `admin`
- `intended_full_name`: text, nullable
- `auth_flow`: text, not null, constrained to `login`, `register`
- `expires_at`: timestamptz, not null
- `consumed_at`: timestamptz, nullable
- `created_at`: timestamptz, not null, default CURRENT_TIMESTAMP

### profiles
- `user_id`: integer primary key references `users(id)`
- `role`: text, not null, constrained to `journalist`, `expert`, `admin`
- `headline`, `organization`, `beat`, `expertise_topics`, `bio`, `city`, `contact_note`: text, nullable
- `created_at`, `updated_at`: timestamptz, not null, default CURRENT_TIMESTAMP

### verification_submissions
- `id`: serial primary key
- `user_id`: integer not null references `users(id)`
- `method_label`: text not null
- `evidence_text`: text not null
- `status`: text not null default `pending`, constrained to `pending`, `approved`, `rejected`
- `reviewer_id`: integer nullable references `users(id)`
- `reviewer_note`: text, nullable
- `created_at`: timestamptz not null default CURRENT_TIMESTAMP
- `reviewed_at`: timestamptz, nullable

### requests
- `id`: serial primary key
- `owner_user_id`: integer not null references `users(id)`
- `title`, `summary`, `topic`, `location`, `deadline_text`: text, with required fields declared in application logic
- `status`: text not null default `open`, constrained to `draft`, `open`, `closed`, `archived`
- `created_at`, `updated_at`: timestamptz not null default CURRENT_TIMESTAMP

### pitches
- `id`: serial primary key
- `request_id`: integer not null references `requests(id)`
- `expert_user_id`: integer not null references `users(id)`
- `message`: text not null
- `proposed_angle`: text, nullable
- `status`: text not null default `pending`, constrained to `pending`, `accepted`, `rejected`, `withdrawn`, `request_closed`
- `created_at`, `updated_at`: timestamptz not null default CURRENT_TIMESTAMP
- unique constraint on `(request_id, expert_user_id)`

### moderation_actions
- `id`: serial primary key
- `admin_user_id`: integer not null references `users(id)`
- `target_type`: text not null constrained to `user`, `request`, `pitch`, `verification`
- `target_id`: integer not null
- `action_type`: text not null constrained to `approve`, `reject`, `flag`, `suspend`, `hide`, `restore`
- `reason`: text not null
- `created_at`: timestamptz not null default CURRENT_TIMESTAMP

## Postgres compatibility notes

- Use `SERIAL` or `GENERATED BY DEFAULT AS IDENTITY` for primary keys.
- PostgreSQL timestamps should use `TIMESTAMPTZ` for audit fields.
- Text columns should remain `TEXT` to preserve compatibility with SQLite strings.
- Unique constraint on `(request_id, expert_user_id)` should be preserved.
